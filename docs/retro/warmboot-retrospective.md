# WarmBoot: A Retrospective

*October 5 - November 15, 2025*

This document preserves the history of the `warm-boot/` directory -- SquadOps' original test harness for multi-agent collaboration. WarmBoot was where we proved agents could build real software, and where we learned everything that shaped the cycle execution architecture (SIP-0064 through SIP-0070) that replaced it.

---

## What Was WarmBoot?

WarmBoot was a self-contained directory (`warm-boot/`) where Max (Lead) and Neo (Dev) would read a PRD, plan tasks, generate source files via LLM, build a Docker image, and deploy a running web app -- all autonomously. Each execution was called a "run" and assigned a sequential ID.

The test subject was always the same: **HelloSquad**, a simple web page served by nginx on port 8080. One PRD, 163+ runs, 105 archived snapshots, six weeks.

## The Journey in Five Eras

### Era 1: Genesis (Runs 001-007, October 5-7)

Everything was new. Run-001 was the first successful WarmBoot -- Max and Neo built a Vue.js/Express app in 50 minutes using Llama 3.1 and Qwen 2.5 via Ollama. Run-002 added version tracking. Run-005 achieved what the wrapup called "100% real agent work" -- zero simulation, zero mocking, real LLM responses producing real files via real RabbitMQ messages. The summary declared it a "paradigm shift from simulation to real autonomous collaboration."

Run-006 rebuilt HelloSquad from scratch in 7.01 seconds. Fastest run ever recorded.

Run-007 was a dedicated RabbitMQ communication verification test with full message-ID-level proof that Max and Neo could actually talk to each other. Both agents used different models (Llama 3.1:8b and Qwen 2.5:7b).

### Era 2: The Pipeline Grind (Runs 008-018, October 16)

A single-day blitz of rapid-fire runs tuning the automated pipeline. SIP-027's event-driven wrapup system came online -- from here forward, every run summary was auto-generated when Neo emitted `task.developer.completed`. Many runs produced 0 artifacts (pure infrastructure testing). By runs 011-017, the archive-build-deploy cycle had settled into a steady rhythm, each run generating 5 files with different SHA256 hashes confirming real (not cached) LLM output.

### Era 3: The Marathon (Runs 067-084, October 17)

An overnight session. The jump from 018 to 067 reflects ~50 runs that only survive as legacy timestamp archives. The simplified PRD was introduced: four features (Welcome Message, Build Information, Real System Data, Framework Transparency). Run-069 was Neo's most architecturally ambitious output -- 8 files including dedicated service modules for a team status dashboard and activity feed. Most runs settled into the standard 5-file SPA pattern, completing in under 2 minutes each.

### Era 4: The Long March to 110 (Runs 100-110, October 19 - November 1)

Two weeks of intermittent runs debugging file paths, Docker builds, and LLM output parsing. Six critical bugs converged to make run-110 THE milestone:

- Health-check app had a duplicate task creation bug
- Max wasn't passing `target_directory` in task requirements
- Neo's file path logic didn't use the target directory
- **The Key Fix**: FileManager wasn't combining directory + file_path (files wrote to bare filenames)
- AppBuilder crashed on None values instead of normalizing to empty string
- FileManager had no defensive checks for empty paths

**Run-110** (November 1) was the first fully successful end-to-end WarmBoot. Real LLM, complete file generation, Docker build, Docker deploy, app live and accessible. The milestone doc declared it "the moment SquadOps moved from working in theory to working in production." Output: 5 files, an 81MB Docker image, a running container on port 8080.

### Era 5: Victory Lap (Runs 119-163, November 2-15)

Post-milestone runs were routine validation. Versions climbed through v0.3, v0.4, v0.6. Run-150 triggered the first "high pulse" warning (29 pulses vs. target of 15) -- a data point that later informed SIP-0070 (Pulse Checks and Verification). Run-155 had a timer bug claiming 4,223 minutes duration.

**Run-163** (November 15) was the final WarmBoot run. HelloSquad v0.6.1.163. Six files generated, including `api.js` for the first time -- the LLM was still evolving its architectural decisions to the very end. The framework had outgrown WarmBoot.

## What Was in the Directory

### Apps (2)

- **hello-squad** -- The flagship. Final version v0.6.1.163. Six files: `index.html`, `styles.css`, `app.js`, `api.js`, `nginx.conf`, `Dockerfile`.
- **application** -- A generic-named clone built under a test ECID. Version v0.6.4. Same structure. A test of the pipeline with a non-hello-squad app name.

### Runs (67 directories, run-001 through run-163)

Each run directory contained:
- A wrapup summary (`warmboot-runXXX-wrapup.md`) -- auto-generated by Max via SIP-027 event-driven coordination
- A logs file (`run-XXX-logs.json`) -- structured event timeline
- A release manifest (`release_manifest.yaml`) -- version, SHA256 hashes, deployment metadata
- Sometimes a requirements doc (`run-XXX-requirements.md`)

### Archive (105 snapshots)

- 69 hello-squad archives spanning v0.1.4 through v0.6.x
- 34 application archives
- 2 hellosquad archives (pre-hyphen naming convention)

Each archive was a full snapshot of the app directory at the time of that run. The earliest entries used Unix timestamps as names (before run numbering was formalized).

### Documentation

- **PRD-001-HelloSquad.md** -- The single PRD that drove all 163+ runs. Never modified.
- **BP-001-HelloSquad.md** -- Business process: five steps from requirements to deployment. Listed Eve and Data as "future" participants.
- **UC-001-HelloSquad.md** -- Use cases for the HelloSquad app.
- **TC-001-HelloSquad.md** -- Five test scenarios (API, HTTP, HTML, integration, E2E).

### Examples

- **warmboot_request.json** -- Sample submission payload showing the late-era simplified "flattened requirements" format.

## Running Themes

**"Consider activating EVE and Data agents for Phase 2."** Every single auto-generated wrapup from run-008 to run-163 ends with this line. Phase 2 never happened within WarmBoot. Eve and Data eventually got their roles in the cycle execution architecture instead.

**SIP-027: The invisible spine.** Every automated wrapup credits SIP-027 Phase 1 event-driven coordination. The pattern of "agent emits completion event, lead generates wrapup" was the seed that grew into the full cycle execution pipeline.

**The PRD that would not die.** PRD-001-HelloSquad.md drove every single run across the entire history. One document, 163+ builds, 105 archives, six weeks of iteration.

## What WarmBoot Taught Us

WarmBoot proved the core thesis: agents can autonomously read requirements, generate code, build containers, and deploy applications. But it also revealed the limitations that drove the architecture forward:

- **Two-agent ceiling**: Only Max and Neo participated. The framework needed structured task planning to coordinate all five agents (SIP-0064).
- **No quality gate**: Neo's code was never tested. Eve never ran. This led directly to the QA build validation handler and test execution (the very feature that retired WarmBoot's era).
- **No observability**: Run wrapups were best-effort. LangFuse integration (SIP-0061) and Prefect flow tracking (SIP-0066) replaced ad-hoc telemetry.
- **One PRD forever**: No mechanism to vary inputs. Cycle request profiles (SIP-0065) solved this.
- **File path chaos**: The "Key Fix" of run-110 (directory + filename combination) was a symptom of agents writing files without a structured artifact vault (SIP-0064).

WarmBoot was the proof of concept. The cycle execution architecture is the production system it proved was possible.
