# SIP-0XXX: Version Bump Hardening — Single-Sourced Strings, Guardrails, and Automated Changelog

**Status:** Proposed
**Authors:** SquadOps Architecture
**Created:** 2026-04-19
**Revision:** 1

## 1. Abstract

`scripts/maintainer/version_cli.py bump` updates the framework version in two files (`pyproject.toml`, `src/squadops/__init__.py`) but the version is *also* hardcoded in at least two operator-facing documents (`README.md`, `CLAUDE.md`). Keeping those in sync is currently a manual step, and the drift is already observable: main's bump from 1.0.1 → 1.0.3 required a separate PR (#56) just to update README and CLAUDE.md, and `CLAUDE.md` still declares *"Framework Version: 1.0.3 (single-sourced via importlib.metadata)"* right next to a string that isn't single-sourced.

There are also no pre-bump guardrails (SemVer format, monotonic ordering, clean working tree), no CHANGELOG.md, and a stale editable install silently reports an old version with no warning — which was caught during SIP-0086 closeout when `pip show squadops` returned 1.0.1 while `pyproject.toml` was at 1.0.3.

This SIP proposes four tightly-scoped improvements to `version_cli.py` plus a new `CHANGELOG.md`:

1. **Single-source version strings** in `README.md` and `CLAUDE.md` via explicit markers so `bump` rewrites them atomically.
2. **Guardrails on `bump`** (SemVer validation, monotonic check, clean-tree check).
3. **Staleness warning** when installed metadata diverges from `pyproject.toml`.
4. **Automated CHANGELOG stub** appended by `bump`, auto-filled with SIP promotions since the last tag.

Each is independently shippable; each closes a concrete friction observed in the 1.0.0–1.0.3 release window.

## 2. Problem Statement

### 2.1 Friction Observed

During SIP-0086 closeout on 2026-04-19, pulling main into the feature branch surfaced the following:

- **`pyproject.toml` and `src/squadops/__init__.py` were at 1.0.3**, bumped as part of unrelated feature commit `bdf5ae7` ("feat: add chat_stream_with_usage…") — breaking the prior convention that version bumps ship in dedicated `chore: bump version to X.Y.Z` commits (observable in the history back through 0.9.16).
- **`README.md` and `CLAUDE.md` were at 1.0.1** until main's PR #56 manually updated them to 1.0.3 — a follow-up PR whose only purpose was to reconcile drift that `version_cli.py` should have prevented.
- **`pip show squadops` returned 1.0.1** because the editable install cache was never refreshed after `bdf5ae7`. `python scripts/maintainer/version_cli.py version` dutifully reported the stale value with no warning.
- **No CHANGELOG**. The release notes for 1.0.0 → 1.0.3 (four bumps, multiple SIP promotions) exist only as scattered commit messages and two README paragraphs. A downstream consumer cannot answer "what changed in 1.0.2?" without `git log`.

### 2.2 Scope of Files That Encode "The Version"

| Location | How it's set | Updated by `bump`? |
|---|---|---|
| `pyproject.toml` `[project].version` | Static string | ✅ yes |
| `src/squadops/__init__.py` `__version__` | Static string (also reads from `importlib.metadata` as fallback) | ✅ yes |
| `README.md` "Current Status: vX.Y.Z" | Hardcoded prose | ❌ no |
| `CLAUDE.md` "Framework Version: X.Y.Z" | Hardcoded prose | ❌ no |
| Installed package metadata (`.dist-info`) | Regenerated by `pip install -e .` | ❌ no (silent staleness) |
| `CHANGELOG.md` | — | Does not exist |

Four places to update, one automation, and one invisible source of truth that can diverge from all of them.

### 2.3 Consequences

- **Docs lie.** A reader landing on README sees "v1.0.1" while the installed package reports "v1.0.3" — or vice versa. Confidence in docs erodes.
- **Release notes live in commit messages.** PR reviewers and downstream users can't answer "what's in 1.0.3" without `git log` spelunking.
- **Typos ship.** `bump "1.0.4 "` (trailing space) or `bump "v1.0.4"` (leading v) would write malformed strings into pyproject.toml with no complaint.
- **Dirty-tree bumps create orphan changes.** A `bump 1.0.4` in the middle of unrelated edits silently mixes them into the version-bump commit.
- **Staleness is invisible.** Nothing warns an operator that their dev environment is reporting a version older than the checkout on disk.

## 3. Design Principles

### 3.1 Version is written by `bump`, read everywhere else

The only file that should *contain* a hardcoded version string is whatever `bump` writes. Anywhere else that needs to display the version should either (a) be rewritten by `bump` via explicit markers, or (b) read from `importlib.metadata` at runtime.

### 3.2 Bump commits are isolated

A version bump is its own commit, with a standardized message, touching only version-carrying files. Mixing a bump into a feature commit is a drift vector.

### 3.3 Guardrails fail fast with actionable messages

A malformed or decreasing version is rejected before any file is written. The error message tells the user what's wrong and what to do.

### 3.4 Observability over enforcement for staleness

`version_cli.py version` should *warn* when installed metadata disagrees with source, not *fail* — a developer mid-task should not be blocked by an admonishment to reinstall.

### 3.5 Changelog is a first-class output of bump

If `bump` doesn't emit a changelog entry, there is no changelog. History shows this is the stable equilibrium. Make the emission automatic and edit-friendly.

## 4. Goals

1. **Zero manual follow-up after `bump`.** A single `version_cli.py bump X.Y.Z` leaves `pyproject.toml`, `__init__.py`, `README.md`, `CLAUDE.md`, and `CHANGELOG.md` all consistent.
2. **Fail fast on malformed or non-monotonic versions.** `bump notaversion` or `bump 0.9.0` (regression) are rejected.
3. **Surface install staleness once, clearly.** The first time `version_cli.py version` runs in an environment where the installed metadata is older than `pyproject.toml`, a single warning line prints.
4. **Ship a CHANGELOG.md populated retroactively** for 1.0.0–1.0.3 as part of this SIP's implementation.
5. **Preserve the convention of bump-as-its-own-commit** by having `bump` refuse to run on a dirty tree (with a `--force` escape hatch).

## 5. Non-Goals

- **Not** replacing SemVer with a different scheme.
- **Not** automating release tagging (`git tag`) or GitHub Releases — those remain maintainer decisions.
- **Not** introducing a build system or packaging change.
- **Not** bumping per-agent versions from `instances.yaml` — that's already handled separately and outside this SIP's scope.
- **Not** changing the acceptance→implemented SIP promotion flow (SIP status is managed by `update_sip_status.py`; this SIP only pulls those transitions into the changelog).

## 6. Design

### 6.1 Single-Sourced Version Strings via Markers

`bump` gets two additional target files. Each line that carries a version string is bracketed by HTML-comment markers so the rewrite is unambiguous and robust to surrounding edits.

**`README.md`:**
```markdown
**Current Status**: <!-- version:readme-status -->v1.0.3<!-- /version:readme-status --> — Production-ready framework with …
```

**`CLAUDE.md`:**
```markdown
**Framework Version**: <!-- version:claude-framework -->1.0.3<!-- /version:claude-framework --> (single-sourced via `importlib.metadata.version("squadops")`)
```

Adding `bump` targets is a small, local change in `version_cli.py`:

```python
MARKER_TARGETS = [
    (README_PATH, "version:readme-status", "v{version}"),
    (CLAUDE_PATH, "version:claude-framework", "{version}"),
]

def rewrite_markers(new_version: str) -> list[str]:
    updated: list[str] = []
    for path, marker, template in MARKER_TARGETS:
        if not path.exists():
            continue
        text = path.read_text()
        pattern = re.compile(
            rf"(<!-- {re.escape(marker)} -->).*?(<!-- /{re.escape(marker)} -->)",
            re.DOTALL,
        )
        new_inner = template.format(version=new_version)
        new_text, count = pattern.subn(rf"\g<1>{new_inner}\g<2>", text)
        if count == 0:
            raise BumpError(f"{path.name}: marker '{marker}' not found")
        if new_text != text:
            path.write_text(new_text)
            updated.append(path.name)
    return updated
```

**Why markers, not heuristic substring replace:** a naive "replace `1.0.3` with `1.0.4`" would corrupt unrelated strings (e.g., `Python 3.11+` or a SIP reference like `v1.0.3`-vintage API docs). Markers make intent explicit and localized.

### 6.2 Guardrails

`bump` validates before touching any file:

```python
_SEMVER_RE = re.compile(r"^\d+\.\d+\.\d+(?:-[A-Za-z0-9.-]+)?$")

def bump_version(new_version: str, *, force: bool = False) -> None:
    if not _SEMVER_RE.match(new_version):
        raise BumpError(
            f"'{new_version}' is not valid SemVer (expected X.Y.Z or X.Y.Z-pre)"
        )

    current = read_pyproject_version()
    if not _is_greater(new_version, current):
        raise BumpError(
            f"New version {new_version} is not greater than current {current}. "
            f"Use a higher version or pass --force if intentional."
        )

    if not force and not _working_tree_clean():
        raise BumpError(
            "Working tree is not clean. Commit or stash first, or pass --force."
        )

    # ... existing pyproject + __init__.py writes ...
    # ... marker rewrites from §6.1 ...
    # ... changelog stub from §6.4 ...
```

The `--force` flag preserves the ability to bump from a messy state when the maintainer knows what they're doing.

### 6.3 Staleness Warning

`get_framework_version()` gains an optional divergence check. Output is a warning line to stderr, not an error, so scripting `version_cli.py version | awk '{print $3}'` still works cleanly.

```python
def get_framework_version() -> str:
    pyproject_version = _read_pyproject_version()
    try:
        installed_version = _pkg_metadata("squadops")["Version"]
    except PackageNotFoundError:
        installed_version = None

    if installed_version and installed_version != pyproject_version:
        print(
            f"WARNING: installed squadops=={installed_version} but "
            f"pyproject.toml declares {pyproject_version}. "
            f"Run `pip install -e .` to refresh.",
            file=sys.stderr,
        )
    return pyproject_version
```

The warning fires at most once per invocation (not per import), keeping it from becoming noise.

### 6.4 Automated CHANGELOG Stub

`bump` prepends a new section to `CHANGELOG.md` following the [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) convention, pre-populated with SIP promotions since the last tag.

**`CHANGELOG.md` structure:**
```markdown
# Changelog

All notable changes to this project will be documented here.
Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

## [1.0.4] — 2026-04-20

### SIPs
- SIP-0086: Build Convergence Loop — Dynamic Task Decomposition (implemented)

### Added
- (edit before release)

### Fixed
- (edit before release)

## [1.0.3] — 2026-04-18
…
```

**Auto-population logic:**

```python
def collect_recent_sip_promotions(since_ref: str) -> list[tuple[str, str]]:

    `implemented` in sips/registry.yaml between since_ref and HEAD.
    """

    # whose status transitioned to "implemented"; return their numbers + titles.
```

The stub is intentionally incomplete — `Added` / `Fixed` sections are placeholders the maintainer edits before cutting the release. The automation captures what's *mechanically* knowable (SIP promotions, version number, date); the human supplies the narrative.

### 6.5 Commit Convention Enforcement (Optional, --commit)

A convenience flag `--commit` creates the version-bump commit immediately with a standardized message:

```bash
python scripts/maintainer/version_cli.py bump 1.0.4 --commit
# Stages pyproject.toml, __init__.py, README.md, CLAUDE.md, CHANGELOG.md
# Creates commit:
#   chore: bump version to 1.0.4
#
#   Updates pyproject.toml, __init__.py, README.md, CLAUDE.md, CHANGELOG.md.
#   See CHANGELOG for included SIP promotions and changes.
```

This preserves the pre-SIP-0086 convention of dedicated bump commits.

## 7. Implementation Plan

### Phase 1: Markers + README/CLAUDE.md integration (1 PR)

1. Add `<!-- version:... -->` markers around the version strings in `README.md` and `CLAUDE.md`.
2. Extend `version_cli.py bump` to rewrite those markers.
3. Unit tests: marker found → rewrite succeeds; marker missing → BumpError with actionable message; multiple markers in one file → all updated.

### Phase 2: Guardrails (1 PR)

1. Add SemVer regex validation.
2. Add monotonic-version check.
3. Add clean-tree check with `--force` escape hatch.
4. Unit tests cover each rejection path + the force override.

### Phase 3: Staleness warning (1 PR, small)

1. Add divergence check to `get_framework_version()`.
2. Unit test: mocked metadata mismatch prints the warning; match does not.

### Phase 4: CHANGELOG (1 PR)

1. Add `CHANGELOG.md` retroactively populated for 1.0.0 through 1.0.3 by reading `sips/registry.yaml` history + existing README release notes.
2. Extend `bump` to prepend a new section with auto-populated SIP list.
3. Unit tests cover: new file creation, existing file prepend (preserves `[Unreleased]`), SIP collection since last tag.

### Phase 5: `--commit` convenience (optional, 1 small PR)

1. Stage and commit the bumped files with the standardized message.
2. Refuse if no changes to bump.

## 8. Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Marker comments in `.md` render visibly in some tools | Cosmetic | HTML comments are hidden by standard Markdown renderers (GitHub, Obsidian). Verified visually before merge. |
| Clean-tree check blocks common dev flow (bump mid-review of unrelated local edits) | Friction | `--force` escape hatch; the default is safer. |
| Auto-populated CHANGELOG entries are wrong or incomplete | Misleading release notes | Maintainer edits before shipping; the stub is intentionally marked `(edit before release)`. |
| Staleness warning becomes noise in CI | Log pollution | Only fires when genuinely divergent; CI typically reinstalls, so the check passes. `--quiet` flag available if needed. |
| Retroactive CHANGELOG for 1.0.0–1.0.3 has gaps | Incomplete history | Document known-incomplete sections; mark as "best-effort reconstruction." |

## 9. Alternatives Considered

### 9.1 Remove version strings from README/CLAUDE.md entirely

Pro: zero drift by construction. Con: the README's "Current Status: vX.Y.Z" is a high-value signal for readers deciding whether to adopt; removing it hides the information. Marker approach preserves the signal and eliminates drift.

### 9.2 Pre-commit hook that rejects commits touching version files without going through `bump`

Pro: strongest enforcement. Con: friction for the common case where a feature PR legitimately shouldn't touch version files at all. Pre-commit hooks also require opt-in setup; a maintainer-CLI path is more reliable.

### 9.3 Fully automated release-please / semantic-release tooling

Pro: industry-standard, generates changelog + bumps from conventional commits. Con: significant tooling adoption, locks in commit message format, overkill for a project where releases are maintainer-gated and infrequent. Revisit post-v2.
